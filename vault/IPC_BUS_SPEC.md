# IPC Bus Specification

> Unix domain socket architecture for inter-pane communication.
> Covers connection lifecycle, NDJSON wire protocol, BusState, lock ordering, event subscriptions.
> Modules: m29_ipc_bus, m30_bus_types | Spec: `WIRE_PROTOCOL_SPEC.md` for frame details
> Plan: `MASTERPLAN.md` V3.2 | Obsidian: `[[Pane-Vortex IPC Bus — Session 019b]]`

## Overview

The IPC bus provides a Unix domain socket for Claude Code instances (spheres) to communicate
without going through HTTP. Each instance connects, performs a handshake, subscribes to
event patterns, and can submit tasks, send cascades, or receive events.

The bus is NDJSON-framed: one JSON object per line, newline-delimited. No binary framing.

## 1. Architecture

```
                          +-----------------+
Claude Instance A ------->|                 |
  (Unix socket)           |   IPC Bus       |  <--- m29_ipc_bus.rs
Claude Instance B ------->|   (BusState)    |
  (Unix socket)           |                 |
Claude Instance C ------->|  Task Queue     |  <--- m30_bus_types.rs
  (Unix socket)           |  Event Router   |
                          |  Sub Manager    |
                          +-----------------+
                                 |
                      +----------+----------+
                      |                     |
               SharedAppState         SharedBusState
               (Arc<RwLock>)          (Arc<RwLock>)
```

### 1.1 Socket Location

```
/run/user/1000/pane-vortex-bus.sock
```

Configurable via `config/default.toml` `[ipc].socket_path`. Permissions: `0o700` (owner only).

### 1.2 Connection Lifecycle

```
Client connects via Unix socket
  |
  v
Client sends: {"type":"handshake","id":"sphere-alpha","version":"2.0"}
  |
  v
Server validates, sends: {"type":"welcome","server_version":"2.0","tick":42}
  |
  v
Client sends: {"type":"subscribe","patterns":["field.*","task.*"]}
  |
  v
Server confirms: {"type":"subscribed","patterns":["field.*","task.*"]}
  |
  v
[Steady state: client sends frames, server routes events]
  |
  v
Client disconnects (EOF) OR server closes (shutdown)
```

### 1.3 Connection Limits

| Limit | Value | Rationale |
|-------|-------|-----------|
| Max connections | 50 | Prevents file descriptor exhaustion |
| Read buffer | 64KB | Per-connection NDJSON line buffer |
| Max line length | 65,536 bytes | Matches HTTP body limit |
| Handshake timeout | 5s | Drop connections that don't handshake |
| Idle timeout | 300s | Drop silent connections (configurable) |

## 2. BusState

```rust
/// m30_bus_types.rs
pub struct BusState {
    pub tasks: HashMap<TaskId, BusTask>,
    pub subscriptions: HashMap<PaneId, Vec<GlobPattern>>,
    pub event_tx: broadcast::Sender<BusEvent>,
    pub cascade_events: VecDeque<CascadeEvent>,
    pub task_counter: u64,
    pub connected_clients: HashSet<PaneId>,
}

pub type SharedBusState = Arc<RwLock<BusState>>;
```

### 2.1 Lock Ordering (CRITICAL)

**AppState BEFORE BusState. Always.**

When a function needs both locks:
```rust
// CORRECT:
let app = app_state.read().await;
let bus = bus_state.write().await;

// WRONG — will deadlock:
let bus = bus_state.write().await;
let app = app_state.read().await;
```

This ordering is enforced by code review and documented in `patterns/CONCURRENCY_PATTERNS.md`.
Every function that acquires both locks MUST be documented in `MODULE_MATRIX.md`.

### 2.2 Event Routing

Events are routed to subscribers via glob-pattern matching:

```rust
/// Match an event type against a subscription pattern
fn pattern_matches(pattern: &str, event_type: &str) -> bool {
    // Glob rules: * matches any segment, ** not supported
    // Examples: "field.*" matches "field.decision", "field.r_update"
    //           "task.*" matches "task.submitted", "task.completed"
    //           "*" matches everything
    glob_match(pattern, event_type)
}
```

Supported patterns:
- `field.*` — all field events
- `task.*` — all task events
- `cascade.*` — all cascade events
- `sphere.{id}.*` — events for a specific sphere
- `*` — everything (firehose mode)

### 2.3 Task Queue

Tasks follow a lifecycle FSM:

```
submitted --> claimed --> completed
    |            |
    |            +--> failed
    |
    +--> expired (TTL)
```

Task targeting types:
- `specific` — routed to a named sphere ID
- `any_idle` — first idle sphere claims it
- `field_driven` — decision engine selects target
- `willing` — any sphere that hasn't opted out (NA consent-aware)

## 3. Event Types

### 3.1 Field Events (generated by tick loop)

| Event Type | Data | Frequency |
|------------|------|-----------|
| `field.tick` | tick number, r, k_mod | Every tick (5s) |
| `field.decision` | FieldDecision JSON | Every tick |
| `field.chimera` | ChimeraState JSON | When chimera detected |
| `field.r_threshold` | r value, direction | When r crosses 0.3/0.8 |
| `field.breathing` | amplitude, period | When breathing detected |
| `field.suggestion` | FieldSuggestion JSON | When generated |

### 3.2 Task Events

| Event Type | Data | When |
|------------|------|------|
| `task.submitted` | task_id, description, target | On submit |
| `task.claimed` | task_id, claimed_by | On claim |
| `task.completed` | task_id, result | On complete |
| `task.failed` | task_id, error | On fail |
| `task.expired` | task_id | On TTL expiry |

### 3.3 Sphere Events

| Event Type | Data | When |
|------------|------|------|
| `sphere.registered` | sphere_id, persona | On registration |
| `sphere.deregistered` | sphere_id | On deregistration |
| `sphere.status_change` | sphere_id, old, new | On status change |

### 3.4 Cascade Events

| Event Type | Data | When |
|------------|------|------|
| `cascade.dispatched` | source, target, brief | On handoff |
| `cascade.acked` | source, target | On ack |
| `cascade.rejected` | source, target, reason | On rejection (NA-P-7) |

### 3.5 Bridge Events

| Event Type | Data | When |
|------------|------|------|
| `bridge.synthex` | thermal_state | On poll |
| `bridge.nexus` | nexus_metrics | On poll |
| `bridge.me` | fitness, health | On poll |
| `bridge.health` | bridge_id, status | On bridge error/recovery |

## 4. Persistence

Bus tasks and events are persisted to `data/bus_tracking.db` via `m36_persistence.rs`.
See `DATABASE_SPEC.md` for schema details.

Key behaviors:
- Tasks written on submit, updated on claim/complete/fail
- Events written in batches (every 10 events or 30s)
- Cascade events written immediately (audit trail)
- WAL mode for concurrent reader/writer access

## 6. Client Binary

`pane-vortex-client` (`src/bin/client.rs`) provides CLI access to the bus:

```bash
# Subscribe and stream events
pane-vortex-client subscribe "field.*"

# Submit a task
pane-vortex-client submit --target any_idle --description "Run cargo test"

# Send a cascade handoff
pane-vortex-client cascade --target tab-4 --brief "Continue with src/api.rs"

# Check bus status
pane-vortex-client status
```

See `WIRE_PROTOCOL_SPEC.md` for full frame definitions.

## 8. Anti-Patterns

- **AP-1:** Acquiring BusState before AppState — guaranteed deadlock
- **AP-2:** Unbounded event broadcast channel — use bounded `broadcast::channel(256)`
- **AP-3:** Blocking in event handler — all handlers must be async, non-blocking
- **AP-4:** Trusting client-provided sphere_id — validate against registered spheres
- **AP-5:** Glob pattern without sanitization — could match unintended event types
